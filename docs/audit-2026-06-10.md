**Date:** 2026-06-10 · **Branch:** `fix/reports-cogs-test-fixtures` (clean tree) · **Method:** automated baselines + five parallel specialized audit passes (realm/RLS boundary, security, business logic, frontend, database) + manual adversarial verification of every high-severity finding. Every HIGH finding below was re-verified by reading the cited code directly; none were refuted.

**Headline:** No confirmed cross-tenant data leak (the extinction-level event) and no SQL injection. RLS coverage is fully green. But there are **12 high-severity findings** — mostly money-math and state-machine bugs that cause direct financial loss (over-refunds), wrongful tenant suspension with no way back, and concurrency races on stock and payment verification — plus one high-severity finding (idempotency cache) whose worst case *is* cross-tenant response disclosure.

## 2. HIGH severity

### H-1. Refunds ignore line discounts — customers can be refunded more than they paid
`apps/api/src/tenant/sale-refunds/sale-refunds.service.ts:284-303` *(verified)*
Refund value uses the **gross** unit price (`orig.unit_price_cents * qty`) and never subtracts the original `discount_cents`. The only cap is per-line quantity; there is no check of refund money against `sale.total_cents - refunded_amount_cents`. A sale of 2 × 10.00 with a 5.00 discount (paid 15.00) can be refunded 20.00 + tax. Direct cash-out loss.
**Fix:** compute per-unit net price (`(line_total_cents)/qty` with remainder handling) and clamp cumulative refunds to the sale total.

### H-2. No path back to `active` — tenants who pay while in grace/suspended stay locked forever
`apps/api/src/admin/billing-tracker/billing-tracker.service.ts:321-327` + `apps/api/src/payment-proofs-shared/payment-proofs.service.ts:279-293` *(verified)*
Verifying a subscription proof marks the invoice `paid` but never touches `tenants.status`. `lifecycleTarget()` only moves forward (`active → grace_period → suspended → cancelled`) and the daily tick `continue`s when no unpaid invoice remains. No admin endpoint mutates tenant status either. The guard's own message — "pay the outstanding invoice to restore write access" — is a promise nothing fulfills.
**Fix:** on subscription-proof verify (and in the daily tick), restore `active` when no unpaid invoices remain; add an admin status override endpoint.

### H-3. `in_review` invoice status is never set — paying tenants get suspended while their proof waits in the queue
`apps/api/src/payment-proofs-shared/payment-proofs.service.ts` (submit) + `billing-tracker.service.ts:152-167` *(verified — `in_review` is only ever read in filters, never written anywhere in `apps/api/src`)*
`submit()` creates the proof without updating the invoice, so it stays `awaiting_payment`/`overdue` and keeps accruing `daysPastDue`. A tenant who paid on time and uploaded a receipt can progress to `grace_period → suspended` purely from verification lag. `docs/billing-flow.md`'s "submitted → verification in progress" state is unreachable.
**Fix:** set invoice `in_review` on proof submit; exclude `in_review` from the lifecycle scan; revert to `awaiting_payment` on reject.

### H-4. P&L double-subtracts discounts — gross profit understated on every discounted sale
`apps/api/src/tenant/reports/pnl/pnl.service.ts:105` *(verified)*
`sales.total_cents` is already net of discount (`sales.service.ts:201-203`), yet the report computes `grossProfit = revenue - discount - tax - cogs`, subtracting `discount_cents` a second time. The per-row breakdown uses the correct formula, so totals and breakdown don't reconcile with each other.
**Fix:** `grossProfit = revenue - tax - cogs` (when revenue includes tax) — and add a reconciliation test totals-vs-breakdown.

### H-5. Virus scanning silently defaults to no-op, including in production
`apps/api/src/env.ts:44` + `apps/api/src/common/virus-scan/virus-scan.module.ts:14-16` *(verified)*
`VIRUS_SCANNER` defaults to `"noop"`; the module factory picks `NoopScanner` (always "clean") unless the env var is explicitly `"clamav"`. No startup guard or warning. A prod deploy that forgets one env var ships with the CLAUDE.md-mandated ClamAV control disabled for all payment-proof receipts and supplier documents.
**Fix:** fail startup when `NODE_ENV=production && VIRUS_SCANNER !== "clamav"` (or require the var explicitly in prod).

### H-6. 17 multi-step writes use `scoped.$transaction(async tx => …)` — a pattern the repo's own comments document as non-atomic
`packages/db/src/tenant.ts:26-38` + e.g. `apps/api/src/tenant/stock-transfers/stock-transfers.service.ts:176,251,311,411`, `stock/stock.service.ts:102`, `purchase-orders/purchase-orders.service.ts:285,387,548`, `supplier-returns/supplier-returns.service.ts:252,340,423`, `held-sales/held-sales.service.ts:184`, `suppliers/suppliers.service.ts:674,767`, `catalog/catalog.service.ts:463`, `payment-proofs-shared/payment-proofs.service.ts:239,480,624` *(verified — pattern and in-repo warning both read)*
The `$extends` wrapper routes **every operation into its own** `basePrisma.$transaction([set_config, query])` batch on a fresh pool connection. `store-credit.service.ts:8-19` documents exactly this ("releasing the lock between statements and silently breaking safety guarantees for concurrent writers"), and sales/sale-refunds/store-credit were already rewritten to `basePrisma.$transaction` + `SET LOCAL`. The remaining 17 sites — including stock movements + `branch_stock` cache writes and payment-proof state changes — get no single-connection atomicity: a crash mid-sequence leaves ledger/cache divergence. (RLS isolation itself is *not* broken — each op still sets the GUC — this is an atomicity/locking bug.)
**Fix:** migrate the 17 sites to the established `basePrisma.$transaction + SET LOCAL app.current_tenant_id` pattern; add a lint rule banning `.$transaction(async` on extended clients.

### H-7. Payment-proof verify/reject race + illegal `refunded → paid` regression
`apps/api/src/payment-proofs-shared/payment-proofs.service.ts:225-253, 467-500` *(verified)*
Status is checked in app code, then `update({ where: { id } })` runs with **no status condition** — concurrent verify+verify double-verifies; verify+reject produces an illegal transition and silently overwrites the first verifier's immutable record (verifier/timestamp/notes). Additionally, `verify()` unconditionally sets the sale `payment_status: "paid"`, so verifying a stale pending proof on a sale that was meanwhile refunded regresses `refunded → paid` while `refunded_amount_cents` still equals the total.
**Fix:** `updateMany({ where: { id, status: "pending" } })` + affected-count check inside the transaction; guard the sale update with its current status.

### H-8. Idempotency cache key is not tenant/user-scoped — cross-tenant response replay
`apps/api/src/common/idempotency.interceptor.ts:54-55` *(verified)*
`cacheKey = idem:{METHOD:route}:{client-supplied UUID}`. Any caller on the same route presenting the same UUID within 24 h receives the **cached response body of the original request** — e.g. another tenant's sale (customer data, amounts) or a signup response containing a live access token. Found independently by two audit passes. Requires knowing/colliding the UUID, but the blast radius is cross-tenant disclosure — the project's defined extinction-level event.
**Fix:** include `tenantId + userId` (or `anon:{ip}` for public routes) in the key; hash the request body and reject same-key/different-payload.

### H-9. Unguarded status transitions on transfers/POs/returns — double send/receive double-moves stock
`apps/api/src/tenant/stock-transfers/stock-transfers.service.ts:284-351,365-462`, `purchase-orders/purchase-orders.service.ts:493-608`, `supplier-returns/supplier-returns.service.ts:398-460`
Same check-then-act class as H-7: two concurrent `send()` calls both observe `draft`, both write movements and both decrement `branch_stock`. Combined with H-10/M-11 (idempotency not concurrency-safe), a client retry can do this without malice.
**Fix:** guarded `updateMany({ where: { id, status: "draft" } })` + count check, inside a real (H-6-fixed) transaction.

### H-10. Stock adjustment computes absolute quantity from a stale read — lost updates diverge cache from ledger
`apps/api/src/tenant/stock/stock.service.ts:83-138`
`beforeQty` is read outside the transaction, `afterQty` computed in app code, and the upsert writes the **absolute** value (not `{ increment: delta }`). Concurrent adjustments (or adjustment racing a sale) lose one delta in `branch_stock.qty_on_hand` while both `stock_movements` rows exist — permanent divergence from the declared source of truth. The negative-stock guard is raceable the same way.
**Fix:** use `{ qty_on_hand: { increment: delta } }` or `SELECT … FOR UPDATE` inside the transaction.

### H-11. Impersonation JWT passed in URL query string
`apps/web/src/app/[locale]/impersonation-handoff/page.tsx:8-19` *(verified)*
The impersonation access token arrives as `?token=…` — it lands in browser history, proxy/server access logs, and potentially `Referer` headers. Compounding: the handoff client (`client.tsx:46-96`) silently installs **any** supplied token into the session on mount with no confirmation or nonce binding (login-CSRF/session-fixation vector — see M-15).
**Fix:** deliver the token via POST + short-lived one-time code exchange, or at minimum URL fragment + immediate `history.replaceState`, plus an interstitial confirmation.

### H-12. Sale/invoice state sync errors are swallowed inside the verification transaction
`apps/api/src/payment-proofs-shared/payment-proofs.service.ts:244-251, 284-291, 491-499, 531-538` *(verified)*
The dependent write (`sale.update → paid/disputed`, `subscriptionInvoice.update → paid`) has a `.catch()` that logs and continues — the proof commits as `verified` while the business record never reflects payment, with no retry or compensation. Swallowing an error inside a Postgres tx also leaves it aborted, making the outer commit fail unpredictably.
**Fix:** let the error propagate and fail the whole verification atomically.

## 5. Verified clean (positives)

- **RLS:** 33/33 tenant tables have `ENABLE` + `FORCE` + FOR-ALL `tenant_isolation` policy with NULLIF-safe `USING` **and** `WITH CHECK` — fail-closed when context unset. Zero gaps. 86/86 isolation tests pass.
- **Realm boundary:** zero raw-Prisma in tenant paths; zero `tenantScoped` in admin paths; zero Prisma in frontends; 100% guard coverage (global `TenantAuthGuard` APP_GUARD + per-controller `AdminAuthGuard`); the 40+ deliberate `adminPrisma` uses inside tenant code (pre-auth, cron, platform tables) all pin `tenant_id` from JWT/job payload.
- **Injection:** every `$queryRaw*`/`$executeRaw*` site parameterized; interpolated fragments are whitelisted enums or zod-validated integers.
- **Money storage:** zero floats in the schema; all money BigInt cents.
- **Stock invariant (runtime code):** all 10 qty-mutating `branchStock` writes have a sibling `stock_movements` create in the same transaction (atomicity caveat: H-6).
- **JWTs:** HS256 pinned, iss/aud enforced, distinct secrets + audiences per realm, refresh rotation with replay detection and family revocation, httpOnly/Secure/SameSite cookies. Tenant MFA-pending flow correctly single-use.
- **Uploads (payment proofs):** 5 MB cap, magic-byte detection, sharp re-encode + EXIF/GPS strip, UUID server-side names, signed URLs (300 s), path-traversal rejected.
- **Secrets:** no `.env` committed; bank numbers AES-256-GCM encrypted, masked to last-4 everywhere, reveal endpoint owner-only + audited.
- **Audit logs:** append-only enforced by triggers **and** `REVOKE` — not just convention.
- **Frontend:** tokens in memory only (no localStorage), no persisted query cache, logical CSS clean, no cross-app contamination, only one (safe, static) `dangerouslySetInnerHTML`.
